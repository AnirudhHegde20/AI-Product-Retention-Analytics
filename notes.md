## 1. Architecture — Warehouse Layering

### The Three Schemas: raw → staging → marts

Think of it like a restaurant:

| Schema | Restaurant Analogy | What it holds |
|--------|-------------------|---------------|
| `raw` | Raw ingredients arriving at back door | Unmodified source data — exactly as scraped/downloaded |
| `staging` | Kitchen cleaning and prep | Standardized, cleaned, type-cast data |
| `marts` | Final dishes served to customers | Business-facing KPIs, cohorts, metrics |

### Key Principle: Raw Data is Immutable
- We **never update or delete** from the raw layer
- It is the source of truth — if anything goes wrong downstream, we reprocess from raw
- If cleaning logic had a bug, raw lets us fix and rerun without re-scraping
- Real-world term: **"immutability of raw data"**

### Why Infrastructure Before Ingestion?
- Schemas must exist **before** data can land in them
- If you run a scraper and the target table doesn't exist → script crashes, data is lost
- Rule: **Infrastructure first → then data flows through it**

## 2. dbt Concepts

### What is dbt?
- dbt (data build tool) is the industry standard for **transforming data inside a warehouse**
- It lets you write modular SQL, track lineage, run tests, and generate documentation
- dbt does NOT move data — it transforms data that's already in the warehouse

### dbt Project Structure
```
ai_saas_dbt/
├── models/
│   ├── staging/     ← cleaning models (one per source table)
│   └── marts/       ← business logic models (KPIs, cohorts)
├── tests/           ← custom data quality tests
├── macros/          ← reusable SQL snippets
├── seeds/           ← small static CSV files loaded into DB
├── snapshots/       ← slowly changing dimension tracking
├── target/          ← auto-generated by dbt, never edit this
└── dbt_project.yml  ← brain of the dbt project
```

### dbt_project.yml
- The configuration file for the entire dbt project
- Tells dbt: where models live, which schema to write them to, how to materialize them
- Uses **YAML format** — indentation = hierarchy (2 spaces per level)

Example:
```yaml
models:
  ai_saas_dbt:        # project name
    +materialized: view   # default: save as view
    staging:
      +schema: staging    # write to staging schema
    marts:
      +schema: marts      # write to marts schema
```

### Materialization: View vs Table

| Type | What it does | When to use |
|------|-------------|-------------|
| `view` | Saves the SQL query — runs on demand, no storage | Staging models — lightweight, always fresh |
| `table` | Physically stores results in DB | Marts — heavy queries, queried often |
| `incremental` | Only processes new rows since last run | Large fact tables, production pipelines |

**Key insight:** Staging models are `view` by default because they're just cleaning raw data — no need to store them physically. Marts models are often `table` because they contain heavy aggregations queried frequently.

## 5. Data Ingestion Concepts

### Raw Table Design — raw.product_hunt_launches

```sql
CREATE TABLE IF NOT EXISTS raw.product_hunt_launches (
    id                SERIAL PRIMARY KEY,
    name              TEXT NOT NULL,
    tagline           TEXT,
    upvotes           INTEGER,
    category          TEXT,
    pricing_type      TEXT,
    launch_date       DATE,
    product_url       TEXT UNIQUE NOT NULL,
    scraped_at        TIMESTAMP DEFAULT NOW()
);
```

### Key Concepts from This Table

**Primary Key (`id`):**
- Uniquely identifies every row in the table
- `SERIAL` means PostgreSQL auto-increments it (1, 2, 3...)
- Every table needs one — prevents ambiguity

**Unique Constraint (`product_url`):**
- No two rows can have the same `product_url`
- Prevents duplicate scrapes — if you run the scraper twice, the same tool won't be inserted twice
- `product_url` is the natural unique identifier for a Product Hunt listing

**NOT NULL:**
- Means this column must always have a value — cannot be empty
- Applied to `name` and `product_url` because a row without these is meaningless

**Audit Metadata (`scraped_at`):**
- Records *when* the scraper ran
- `DEFAULT NOW()` means PostgreSQL automatically fills this in with the current timestamp
- Used for: tracking data freshness, debugging issues, understanding pipeline history
- Real-world term: **audit metadata** — data about your data

### Why We Design Raw Tables Before Scraping
- The raw table is the **landing zone** for scraped data
- If the table doesn't exist when the scraper runs → script crashes, data is lost
- Rule: **Infrastructure first → then data flows through it**

## 6. Ingestion Script Concepts

### The 6-Station Pipeline Pattern
Every ingestion script follows this pattern:

```
1. Load credentials → 2. Connect to DB → 3. Define query
       → 4. Call API → 5. Parse + Insert → 6. Commit + Close
```

### Key Concepts

**Transactions & commit():**
- All inserts are temporary until `commit()` is called
- Like a Google Doc — changes exist in memory until you save
- If script crashes before `commit()` → all inserts are lost
- `commit()` = permanently write to disk

**Upsert Pattern (`ON CONFLICT DO NOTHING`):**
- If a row with the same `product_url` already exists → skip it silently
- Prevents duplicates when scraper runs multiple times
- Real-world term: **upsert** (update + insert combined)
- Running scraper twice → same 20 rows attempted, 0 duplicates inserted ✅

**GraphQL vs REST API:**
- REST: gives you everything, you filter what you need
- GraphQL: you ask for exactly what you need, nothing more
- Less wasted data, faster responses
- Product Hunt uses GraphQL

**Credential Hygiene:**
- Never hardcode API tokens in scripts
- Store in `.env` file, load with `python-dotenv`
- Add `.env` to `.gitignore` so it never gets pushed to GitHub
- Bots constantly scan GitHub for exposed credentials

**psycopg2 Concepts:**
- `conn` = the connection to PostgreSQL (like a phone call)
- `cur` = the cursor (like a pen to write SQL commands)
- Always close both after use to free up resources

### Interview Talking Points
- "I used an upsert pattern with ON CONFLICT DO NOTHING to make the pipeline idempotent — running it multiple times never creates duplicates."
- "I stored credentials in a .env file and gitignored it — never committed secrets to GitHub."
- "I used GraphQL to pull exactly the fields I needed, mapping them directly to my raw table schema."

## 7. dbt Models — Staging Layer

### What is a Staging Model?
- Takes raw data as input, cleans and standardizes it
- One staging model per source table
- Stored as a **view** (not a table) — lightweight, always fresh
- Never contains business logic — just cleaning

### Staging Model Structure
```sql
WITH source AS (
    -- Pull from raw using dbt source reference
    SELECT * FROM {{ source('raw', 'product_hunt_launches') }}
),
cleaned AS (
    SELECT
        TRIM(LOWER(name)) AS name,   -- standardize text
        CASE
            WHEN upvotes >= 700 THEN 'High Traction'
            ELSE 'Low Traction'
        END AS traction_tier          -- derive new columns
    FROM source
)
SELECT * FROM cleaned
```

### Key dbt Concepts

**`{{ source('raw', 'product_hunt_launches') }}`**
- dbt's way of referencing a raw table
- Defined in `sources.yml` — the source contract
- dbt tracks this as a dependency in the lineage graph

**sources.yml — The Source Contract**
- Tells dbt which raw tables exist and where to find them
- Without this, dbt doesn't know what `{{ source(...) }}` means
- Also enables freshness checks later

**SQL Transformations Used:**
- `TRIM()` — removes leading/trailing whitespace
- `LOWER()` — converts text to lowercase for consistency
- `CASE WHEN` — if/else logic to derive new column values

### Running dbt
```bash
cd ai_saas_dbt   # must be inside dbt project folder
dbt run          # compiles + executes all models
```

Output to look for:
```
1 of 1 OK created sql view model analytics_staging.stg_product_hunt_launches
PASS=1 WARN=0 ERROR=0
```

### Interview Talking Points
- "Staging models are views — they don't store data physically, they always reflect the latest raw data."
- "I used dbt source contracts to declare raw tables — this lets dbt track lineage and run freshness checks."
- "Staging is purely for cleaning — no business logic. Business logic lives in marts."

## 8. dbt Tests & Data Observability

### What are dbt Tests?
- Automated checks that run after every `dbt run`
- Catch data quality issues before they reach dashboards
- Defined in `schema.yml` alongside your models

### Built-in dbt Tests
| Test | What it checks |
|------|---------------|
| `not_null` | Column has no NULL values |
| `unique` | No duplicate values in column |
| `accepted_values` | Only specified values exist in column |
| `relationships` | Foreign key integrity between tables |

### schema.yml Structure
```yaml
version: 2
models:
  - name: stg_product_hunt_launches
    columns:
      - name: product_url
        tests:
          - not_null
          - unique
      - name: traction_tier
        tests:
          - accepted_values:
              arguments:
                values: ['High Traction', 'Medium Traction', 'Low Traction']
```

### Key Commands
```bash
dbt run        # compile + execute models only
dbt test       # run tests only
dbt build      # run models + tests together (use this in production)
```

### Silent Failures
- Pipeline runs without errors but data is wrong
- Example: API returns 0 rows → dashboard shows blank charts
- dbt tests catch this by asserting expected conditions
- If test fails → loud error, not silent failure

### Defense in Depth
Two layers of data protection:
1. **Database constraints** (`NOT NULL`, `UNIQUE`) — reject bad data at insert time
2. **dbt tests** — catch bad data that slips through to models

### Views vs Tables — Refresh Behavior
- Views don't store data — they run the SQL query on demand
- To see updated raw data in a view → run `dbt run` to recompile
- Tables store data physically → need explicit refresh

### Interview Talking Points
- "I implemented defense in depth — database constraints at the raw layer and dbt tests at the staging layer, so bad data gets caught at two different points."
- "dbt build runs models and tests together — if any test fails, the pipeline fails loudly instead of silently producing wrong data."
- "I deliberately simulated a silent failure to demonstrate why observability tests matter."

## 9. dbt Models — Marts Layer

### What is a Marts Model?
- Answers real business questions — not just cleaning
- Built on top of staging models, never directly on raw
- Stored as **tables** (not views) — heavy aggregations queried frequently
- Named with `fct_` (fact) or `dim_` (dimension) prefix

### Fact vs Dimension Tables
| Type | Prefix | What it holds | Example |
|------|--------|--------------|---------|
| Fact | `fct_` | Metrics and measurements | `fct_launch_performance` |
| Dimension | `dim_` | Descriptive attributes | `dim_categories` |

### ref() vs source()
| Function | Used for | Example |
|----------|---------|---------|
| `{{ source() }}` | Raw tables outside dbt | `{{ source('raw', 'product_hunt_launches') }}` |
| `{{ ref() }}` | dbt models (staging or marts) | `{{ ref('stg_product_hunt_launches') }}` |

**Key insight:** dbt uses `ref()` and `source()` to automatically build the lineage graph and determine build order.

### Lineage Graph
```
raw.product_hunt_launches 
    → stg_product_hunt_launches 
        → fct_launch_performance 
            → fct_category_performance
```
- Built automatically from `ref()` and `source()` calls
- Visualized with `dbt docs generate && dbt docs serve`
- Shows data flow across entire pipeline

### marts Models Built
**fct_launch_performance** — scores every Product Hunt launch:
- `success_score` — 0-100 score based on upvotes
- `days_since_launch` — how old is the product
- `is_top_performer` — boolean flag for high traction tools
- `launch_month` — for time-based analysis

**fct_category_performance** — aggregates by category:
- `avg_success_score` — average performance per category
- `top_performer_pct` — % of launches that are top performers
- `total_launches` — competition level per category

### Key Business Insights from Data
- Augmented Reality + Open Source: 100% top performer rate
- Artificial Intelligence: 0% top performer rate — crowded, hard to stand out
- Productivity: most launches (4) but only 25% top performer rate

### dbt Docs Commands
```bash
dbt docs generate   # generates documentation + lineage graph
dbt docs serve      # opens browser at localhost:8080
```

### Interview Talking Points
- "Marts models reference staging via ref() — dbt uses this to track lineage and enforce build order automatically."
- "I materialized marts as tables because they contain heavy aggregations queried frequently, unlike staging views."
- "The lineage graph shows the full data flow from raw through staging to marts — generated automatically by dbt."

## 10. Yelp Ingestion — Big Data Concepts

### Why Yelp?
Yelp's dataset is real behavioral data — actual users, actual timestamps, actual interactions. It maps perfectly to SaaS product analytics:

| Yelp Concept | SaaS Equivalent |
|---|---|
| User | App user |
| Review | Engagement event (feature used) |
| Review date | Event timestamp |
| First review date | User signup date |
| Last review date | Last active date |
| Star rating | Engagement quality score |

### Batch Processing
Loading 6GB+ of data all at once would crash a local database. Instead we use **batch processing** — loading data in chunks of 10,000 rows at a time, up to a maximum of 500,000 rows.

```python
CHUNK_SIZE = 10_000   # rows per trip (like carrying 500 books at a time)
MAX_ROWS   = 500_000  # total rows to load (we stop here)
```

This is a fundamental big data concept — never try to move everything at once.

### Natural Keys vs Surrogate Keys
Yelp data comes with its own unique IDs (`review_id`, `user_id`). We use these directly as primary keys instead of generating new ones with `SERIAL`. This is called a **natural key** — a unique identifier that comes from the source system itself. Using natural keys keeps the pipeline idempotent — reloading data never creates duplicates because the same review always has the same ID.

### Yelp Staging Models — Key Transformations

**stg_yelp_reviews** derives:
- `engagement_score` — combines stars + useful + funny + cool into one signal
- `sentiment` — Positive (4-5 stars), Neutral (3 stars), Negative (1-2 stars)
- `review_month` — truncates date to month for cohort grouping

**stg_yelp_users** derives:
- `user_segment` — Power User (100+ reviews), Regular (20+), Casual (5+), New (<5)
- `influence_tier` — Influencer (100+ fans), Active (10+), Standard
- `cohort_month` — the month a user first joined, used for retention cohort grouping

## 11. Retention Cohort Analysis

### What is a Retention Cohort?
A cohort is a group of users who share a common starting point — in our case, the month they first joined. Cohort retention analysis asks: "Of everyone who joined in month X, what percentage were still active in month X+1, X+2, X+3?"

This is the single most important metric for any SaaS product team.

### How fct_retention_cohorts Works
The model does three things in sequence. First it joins users with their reviews to find every month each user was active. Then it calculates `months_since_joining` — how many months after joining each activity happened. Finally it counts active users per cohort per month and divides by the cohort size to get a retention rate.

```
Month 0 → 100% (everyone who just joined)
Month 1 → 29%  (only 29% came back next month)
Month 3 → 21%  (stabilizing)
Month 6 → 21%  (slight recovery — some users returned)
```

### Key Insight from the Data
70% of users churned within their first month. This is a **retention cliff** — a steep drop-off in month 1 that then flattens out. In SaaS, this signals that the product isn't delivering enough value in the onboarding phase to bring users back. The right response is root cause analysis first — understand *why* users leave before jumping to solutions like discounts or new features.

### months_since_joining = 0
This is how we calculate cohort size. At month 0, every user in the cohort is counted — that becomes the denominator for all future retention rate calculations.

### Interview Talking Points
- "I built retention cohorts by joining 500k user records with 500k review events in a dbt mart model, calculating month-over-month retention rates for every cohort going back to 2005."
- "The data showed a 70% month-1 churn rate — a classic retention cliff that signals weak onboarding value delivery."
- "I used batch processing with a chunk size of 10,000 rows to safely load 500k records into PostgreSQL without overwhelming local memory."

## 12. CI/CD with GitHub Actions

### What is CI/CD?
- **CI (Continuous Integration):** Every code push automatically triggers a build + test run
- **CD (Continuous Deployment):** Automatically deploy if tests pass
- In our project we implement CI — every push to `main` runs `dbt build`

### Why CI/CD Matters
Without CI/CD, broken code can silently merge into main. With CI/CD, if any dbt test fails → the pipeline fails loudly → bad data never reaches production.

### How GitHub Actions Works
- Special file at `.github/workflows/dbt_ci.yml` tells GitHub what to do on every push
- GitHub spins up a **fresh, empty environment** every time — nothing from your local machine carries over
- This is why we need to recreate schemas and raw tables in the workflow

### Our CI/CD Pipeline — 7 Steps
1. Checkout code from GitHub
2. Set up Python 3.9
3. Install dbt-postgres
4. Create `profiles.yml` pointing to CI database
5. Create schemas (raw, staging, marts) — fresh environment needs this
6. Create raw tables — dbt needs these to exist before running
7. Run `dbt build` — compiles all models + runs all 31 tests

### dbt vs Airflow
| Tool | Role | When to use |
|------|------|-------------|
| dbt | Transforms data **inside** the warehouse | SQL modeling, tests, docs |
| Airflow | **Orchestrates** entire pipeline end to end | Scheduling, triggering scripts, dependencies |
| GitHub Actions | **CI/CD** — quality gate on every push | Automated testing, lightweight scheduling |

### Interview Talking Points
- "I set up CI/CD with GitHub Actions so every push to main automatically runs dbt build — if any of the 31 data quality tests fail, the pipeline fails loudly instead of silently."
- "The CI environment spins up a fresh PostgreSQL instance each time, which means I had to recreate schemas and raw tables as part of the workflow — same infrastructure-first principle as local development."
- "I understand the difference between dbt (transformation), Airflow (orchestration), and GitHub Actions (CI/CD) — they solve different problems in the data stack."
